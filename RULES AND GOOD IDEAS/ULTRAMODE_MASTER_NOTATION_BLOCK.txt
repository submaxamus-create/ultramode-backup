SECTION 0 — GLOBAL MASTER RULES
==============================================================================

0.1 — PATCHER COMPATIBILITY FIRST  
- Every AI-dumped block MUST begin with:
  - `FILENAME: <descriptive_name.ext>`  
  - `PATCHER_COMPATIBILITY: REQUIRED`  
  - `PATCHER_HEADER_VERSION: <x.y>`  
- No block is valid for use unless it starts with a patcher header.  
- Patcher header is considered part of the NOTATION and must never be removed.  
- Patcher uses FILENAME to:
  - Identify the target file or logical unit.  
  - Safely match and apply patches.  
  - Avoid ambiguity between multiple blocks.

0.2 — FILENAME IS MANDATORY  
- Every AI-dumped block MUST declare a FILENAME.  
- FILENAME must be:
  - Stable, human-readable, and unique within its context.  
  - Either the real file path or a canonical logical name.  
- If the block describes a script, FILENAME should match the script:
  - Example: `FILENAME: um.py`  
  - Example: `FILENAME: modules/mandatory/um_gamepad.py`  
- If the block is a pure notation/meta file:
  - Example: `FILENAME: ULTRAMODE_MASTER_NOTATION_BLOCK.txt`

0.3 — PATCHER-SAFE BLOCK STRUCTURE  
- Order of any AI-dumped block:
  1. Patcher header (FILENAME + PATCHER_COMPATIBILITY + PATCHER_HEADER_VERSION)  
  2. High-level title and version  
  3. Sections with stable headings  
  4. Optional PATCH BLOCK definitions  
- Sections and headings must be stable so the patcher can:
  - Locate sections by name.  
  - Append new content safely.  
  - Avoid rewriting or deleting existing content.

0.4 — APPEND-ONLY NOTATION  
- All NOTATION is append-only.  
- No destructive edits to previous content.  
- New rules are added as new sections or sub-sections.  
- If a rule is superseded, the new rule must explicitly state:
  - Which rule it supersedes.  
  - Why it changes behavior.  
- Patcher must never remove old rules; it only adds new ones.

SECTION 1 — ULTRAMODE PHILOSOPHY & GOVERNANCE
==============================================================================

1.1 — ULTRAMODE IS A TREE  
- UltraMode is a living tree of modules, roots, branches, and leaves.  
- `um.py` is the ROOT.  
- Mandatory modules are the TRUNK.  
- Optional modules are BRANCHES.  
- Features, UI, logs, and user interactions are LEAVES.  
- All growth is append-only, never destructive.  
- All modules must be able to fall back to the root (`um.py`) for safety.

1.2 — ULTRAMODE IS CONTROLLER-FIRST  
- Every module must be fully operable with a controller.  
- Keyboard/mouse are secondary.  
- Gamepad input is centralized and never reimplemented.  
- All UI must be navigable with D-pad, analog, A/B/X/Y, Start/Back.

1.3 — ULTRAMODE IS PATCH-SAFE  
- All scripts must contain PATCH BLOCKS.  
- All modifications must be idempotent.  
- No script may overwrite user data without explicit confirmation.  
- All patches must be reversible or non-destructive.

1.4 — ULTRAMODE IS LOG-DRIVEN  
- Every module logs actions to its own category.  
- Logs are append-only.  
- Logs are used to generate developer documentation automatically.  
- Logs are used to teach future modules how to behave.

1.5 — ULTRAMODE IS SELF-DOCUMENTING  
- Every module must contain a NOTATION section.  
- NOTATION grows over time.  
- NOTATION is the “DNA” of the module.  
- NOTATION is never deleted, only appended.

SECTION 3 — GAMEPAD ARCHITECTURE (CANONICAL)
==============================================================================

3.1 — GAMEPAD IS CENTRALIZED  
- Only one module handles gamepad input: `mandatory/um_gamepad.py`.  
- No module may read joystick events directly.  
- No module may use pygame or evdev directly.  
- All modules must call UMGamepad’s public API.

3.2 — GAMEPAD PUBLIC API (LOGICAL CONTRACT)  
UMGamepad exposes:
- `get_input()` — returns a unified input event.  
- `wait_for_input()` — blocking wait for next event.  
- `is_pressed(button)` — check button state.  
- `consume()` — clear input buffer.  
- `map(button, action)` — remap controls (future).  
- `rumble()` — haptic feedback (future).  
- `analog()` — analog stick values (future).  
- `hotkeys()` — global shortcuts (future).

3.3 — GAMEPAD EVENT MODEL  
Events are normalized:
- `UP`, `DOWN`, `LEFT`, `RIGHT`  
- `A`, `B`, `X`, `Y`  
- `START`, `BACK`  
- `LB`, `RB`  
- `LT`, `RT`  
- `L3`, `R3`  
- `ANALOG_MOVE` (future)

3.4 — GAMEPAD INTEGRATION IN MODULES  
Every module must:
- Import UMGamepad from mandatory.  
- Use UMGamepad for all input.  
- Never poll input manually.  
- Never block without using UMGamepad’s wait functions.  
- Provide controller-friendly UI navigation.

3.5 — GAMEPAD FALLBACK  
If gamepad fails:
- UMGamepad falls back to keyboard.  
- UMGamepad logs the fallback.  
- UMGamepad exposes the same API.  
- Modules do not need to change.

SECTION 9 — GLOBAL COLLABORATION / HANDOFF RULES (APPENDED 2026-03-21)

9.1 — SHORT DATE/TIME NOTATION IN CHAT
- All future AI chat replies for project work should include a very short date/time notation.
- Purpose:
  - Help the user avoid getting lost in long chats.
  - Make handoff timing easier to understand.
  - Reduce confusion when chats are reopened later.

9.2 — ASK BEFORE DUMPING FILES OR LONG TEXT
- AI should not dump long scripts, long sidecars, or large explanations without asking first,
  unless the user directly asked for the dump in that message.
- Preferred behavior:
  - keep chat short
  - ask first
  - one step at a time
- This rule is global across UM, sidecars, and related project chats.

9.3 — FILE HANDOFF RULE
- Files handed back to the user should include a version in the filename when practical.
- Example:
  - um_v9.1.1_r2.py
  - discover_plugins_v0.3.py
- Purpose:
  - Reduce accidental overwrite confusion.
  - Reduce testing of wrong duplicate copies.
  - Make handoff state clearer.

9.4 — TXT SIDECAR / LOG RULE
- Sidecars, logs, handoff notes, and rules files should prefer .txt extensions.
- Reason:
  - Easier upload back into AI chats.
  - Easier cross-AI handoff.
  - Less confusion than special-purpose extensions.
- Examples:
  - *_log.txt
  - *_help_log.txt
  - *_rules_log.txt
  - *_thinktank_log.txt

9.5 — GLOBAL HANDOFF / LAW-BOOK IDEA
- UltraMode should maintain a durable global handoff / rules file for future AI.

  - Preserve rules, structure, safety expectations, notation standards, and learned lessons.

SECTION 10 — GUI / WINDOW / RESIZE RULES (APPENDED 2026-03-21)
==============================================================================

10.1 — NATIVE WINDOW FRAME TERMINOLOGY
- The "_ square X" top-right controls are the native OS window frame / native window decorations / window chrome.
- Important:
  - These are not custom pygame-drawn buttons unless explicitly implemented.
  - If a script needs that behavior, prefer a normal decorated resizable pygame window first.

10.2 — PRESERVE WORKING ENGINE, PORT VISUALS CAREFULLY
- When improving GUI appearance, future AI should prefer:
  - old engine + new paint
- Meaning:
  - preserve working input loop
  - preserve working gamepad integration
  - preserve working launcher behavior
  - port visual styling carefully
- Do not rewrite working runtime core just to chase nicer visuals.

10.3 — SMART DEFAULT WINDOW SIZE
- GUI modules should aim to open at a smart default size that fits content nicely.
- Resizing should not merely stretch the window.
- Resizing should work with:
  - UI scale
  - font scale
  - panel spacing
  - bubble sizes
  - padding
- Goal:
  - adaptive layout
  - readable text
  - reduced clipping
  - reduced cramped appearance

10.4 — ZOOM / UI SCALE AS RESIZE BASE
- If a module already has zoom or UI scale concepts, future AI should consider using that
  as the base system for resize behavior.
- Better idea:
  - one scale factor drives fonts, padding, borders, and panel metrics together
- Avoid separate sloppy one-off resize hacks.

10.5 — GUI FAMILY LEARNING SOURCES
- Future AI may learn visual direction from:
  - umpatcher.py
  - patch_source_loader_gui_v7.py
- These are learning references, not forced clones.
- Good takeaways include:
  - richer dark GUI family feel
  - stronger framed sections / bubbles
  - cleaner header/footer presence
  - controller-friendly navigation
  - readable highlighted states
- Future AI must preserve module identity and should not blindly copy appearances.

SECTION 12 — MISTAKES LEARNED / PATCH SAFETY (APPENDED 2026-03-21)
==============================================================================

12.1 — MISSING HELPER CHAINS
- Recent work showed that large GUI rewrites can accidentally omit helper functions
  and crash at runtime.
- Future AI must verify:
  - helper functions exist
  - functions are defined before use
  - status/color/display helpers are present
  - event-loop helpers are preserved
- Do not assume syntax-checking alone proves runtime safety.

12.2 — DUPLICATE FILE CONFUSION
- Testing wrong duplicate filenames caused confusion about whether a fix had worked.
- Future AI should:
  - prefer versioned filenames for handoff
  - warn about duplicate copies
  - help the user keep one active live script when possible

12.3 — DO NOT BREAK GAMEPAD TO IMPROVE SKIN
- Gamepad and launcher flow are core behavior.
- Visual modernization must not break controller handling.
- If there is a conflict between visual ambition and working input, preserve working input.

12.4 — APPEND-ONLY RULE FOR LESSONS
- Important mistakes and lessons should be preserved here rather than forgotten.
- Future AI should append new lessons when they materially improve patch safety.

  - Use one clear vertical menu list with one selected row at a time.
  - Keep the main action near the top, ideally first, if the module has an

      2. left-side actions/menu
      3. right-side details/preview/recommendations
      4. footer hints / controls / ticker / status
  - Other modules do NOT need to be pixel-identical, but they should feel like
    members of the same family.

  - Python 3 with shebang first line.
  - pygame for windowing, rendering, keyboard, and joystick/gamepad events.
  - pygame.joystick for controller support.
  - Standard library support such as os, sys, time, shutil, traceback,
    tempfile/path handling, and atomic write helpers where relevant.

ULTRAMODE NOTATION BLOCK
CATEGORY: PATCHING_ARCHITECTURE
TYPE: PATCH_VERIFICATION_AND_SECTION-BASED_PATCHING_RULESET
STATUS: ACTIVE IDEA / STRONG RECOMMENDATION
PRIORITY: VERY HIGH
===============================================================================

TITLE:
PATCHER MUST VERIFY REAL CHANGE AFTER PATCHING AND MOVE TOWARD SECTION-BASED PATCHING

CORE IDEA:
After a patch is applied, the patcher should not merely trust that the write
operation succeeded. It should immediately re-read the target file and confirm
that the intended change truly landed in the live file content.

At the same time, the patcher should evolve away from fragile single-line
anchor patching and toward section-based patching, where clearly bounded
sections or named regions are targeted instead of one brittle line.

-------------------------------------------------------------------------------
PART 1: PATCH VERIFICATION / SELF-CHECK
-------------------------------------------------------------------------------

PROBLEM THIS SOLVES:
Single-line or fragile anchor patching can appear to apply successfully while
failing to change the real behavior of the file. A patch may report success,
touch the file, or alter formatting slightly, while the actual intended logic
does not land where it should. This creates a false "patch applied" feeling.

IMPORTANT LESSON:
A patch operation is not truly successful just because:
- the file was written
- the file timestamp changed
- file size changed
- a success message appeared
- a backup was created

A patch should only be considered successful if the intended content can be
found and validated in the resulting live file after patching.

RECOMMENDED PATCHER BEHAVIOR:
1. Read target file before patching.
2. Store pre-patch content snapshot.
3. Apply patch.
4. Re-read the file from disk after write.
5. Compare post-patch content against expected outcome.
6. Confirm that the intended inserted/replaced/section content now exists.
7. Confirm that the old content changed in the expected way.
8. Only then report true success.

WHY FILE-SIZE CHECKING ALONE IS NOT ENOUGH:
Checking whether a file became larger or smaller may sometimes be useful as a
secondary clue, but file-size change alone is not reliable proof of a correct
patch. A file may grow, shrink, or stay nearly the same size while still being
wrong. Therefore size delta should be treated only as supplemental diagnostic
information, never as the main validation method.

SMART VALIDATION IDEAS:
The patcher should eventually support several layers of self-check logic:

A. CONTENT PRESENCE CHECK
- Verify that the new intended text exists in the file.

B. OLD CONTENT CHANGE CHECK
- Verify that the original targeted content was changed, removed, or replaced
  exactly as expected.

C. SECTION VALIDATION
- If patching by named section, re-read that exact section and confirm that
  the section now matches the expected result.

D. SIZE DELTA DIAGNOSTIC
- Optionally record:
  - bytes before
  - bytes after
  - size delta
- This is useful for human review, but should not determine success by itself.

E. HASH / SNAPSHOT IDEA
- Future idea: generate before/after hashes or fingerprints for the target
  section or whole file so patch results can be audited more reliably.

F. DIFF PREVIEW / DIFF CONFIRMATION
- Future idea: show a before/after diff of the live result after patching, not
  just the intended preview before patching.

BEST PRACTICE RULE:
The patcher should verify semantic success, not just write success.

This means:
- Did the intended code/text actually land?
- Did it land in the correct section?
- Did the target file meaningfully change?
- Did the patch avoid duplicate or partial geometry/code blocks?
- Did the result match the design intent?

RECOMMENDED SUCCESS STATES:
Instead of one vague "Patch Applied" message, future versions may use clearer
outcomes such as:

- VERIFIED_SUCCESS
  Patch applied and live file content matches expected change.

- PARTIAL_SUCCESS
  File changed, but validation is incomplete or uncertain.

- WRITE_ONLY_WARNING
  File was written, but intended content could not be confirmed.

- VALIDATION_FAILED
  Patch attempt completed, but live file does not match expected result.

- NO_MEANINGFUL_CHANGE
  Operation ran, but file content is effectively unchanged.

WHY THIS MATTERS FOR ULTRAMODE:
UltraMode is modular, evolving, and often edited through AI-assisted workflows.
This increases the risk of:
- fake success messages
- stale assumptions
- indentation damage
- duplicate blocks
- partial edits
- patching the wrong live content
- user thinking a fix landed when it did not

Therefore the patcher must become increasingly self-checking, skeptical, and
honest about what really changed.

FUTURE DIRECTION:
Preferred long-term direction is:
- move away from fragile single-line anchor logic
- move toward section-based patching
- validate exact live-file results after apply
- centralize backups and validation reporting
- record patch verification details in sidecar notation/logging

THINKTANK NOTE:
A truly intelligent patcher should behave less like a blind text replacer and
more like a cautious file surgeon:
- inspect before
- patch carefully
- inspect after
- prove the result
- warn if uncertain

-------------------------------------------------------------------------------
PART 2: SECTION-BASED PATCHING / REPLACE_SECTION
-------------------------------------------------------------------------------

PROBLEM THIS SOLVES:
Single-line anchor patching is fragile. It can fail when:
- spacing changes
- indentation changes
- comments are added
- duplicate lines exist elsewhere
- AI rewrites nearby code
- a line looks matched but is not the true intended target
- patch reports success while the real live behavior does not change

WHAT A FRAGILE ANCHOR IS:
A fragile anchor is a patch target based on:
- one exact line
- one short text fragment
- one repeated string
- one nearby formatting pattern
- one assumption that the target file still looks exactly the same

This is dangerous in evolving AI-assisted files.

SECTION-BASED PATCHING CONCEPT:
A section-based patcher should work with clear structural units, such as:
- named sections
- bounded code blocks
- marked regions
- notation blocks
- function bodies
- class bodies
- UI layout blocks
- configuration sections
- grouped menu geometry sections

EXAMPLE SECTION THINKING:
Instead of saying:
"insert after this one line"

the patcher should be able to say:
"replace the content of the CONFIRM_MODAL_LAYOUT section"
or
"insert this block inside the FOOTER_LAYOUT section"
or
"replace everything between SECTION START and SECTION END markers"

PREFERRED SECTION TYPES:
1. EXPLICIT NAMED SECTION MARKERS
   Example idea:
   - SECTION: CONFIRM_MODAL_LAYOUT_BEGIN
   - SECTION: CONFIRM_MODAL_LAYOUT_END

2. FUNCTION-LEVEL TARGETING
   Example:
   - replace function body of build_confirm_modal()

3. NOTATION OR COMMENT-BOUNDED REGIONS
   Example:
   - replace code between special comment markers

4. FUTURE STRUCTURAL PARSING
   Example:
   - Python-aware parsing of function/class boundaries
   - safer but more advanced future direction

RECOMMENDED PATCH TYPES TO SUPPORT:
- REPLACE_SECTION
- INSERT_IN_SECTION
- APPEND_TO_SECTION
- PREPEND_TO_SECTION
- DELETE_SECTION
- VALIDATE_SECTION
- EXTRACT_SECTION
- DIFF_SECTION

MOST IMPORTANT NEW PATCH TYPE:
REPLACE_SECTION

DEFINITION:
REPLACE_SECTION should locate a clearly defined target section and replace only
that section's internal content, or replace the whole bounded block, depending
on rule design.

WHY REPLACE_SECTION MATTERS:
It is much safer than fragile one-line replacement because:
- the target is larger and more meaningful
- the patch intent is clearer
- validation is easier afterward
- duplicate/partial block errors become easier to detect
- UI/layout cleanup becomes more manageable
- self-patching becomes less reckless

RECOMMENDED REPLACE_SECTION WORKFLOW:
1. Read live target file.
2. Locate section boundaries.
3. Confirm exactly one valid intended section match.
4. Snapshot old section content.
5. Replace section content with new block.
6. Write file.
7. Re-read file.
8. Re-extract section.
9. Verify section now matches expected new result.
10. Report VERIFIED_SUCCESS only if section truly matches.

IMPORTANT VALIDATION RULE:
A section patch should fail loudly if:
- section start marker is missing
- section end marker is missing
- multiple ambiguous matches exist
- section boundaries are corrupted
- resulting section content does not match expected text
- file write occurred but intended section result is not proven

WHY THIS HELPS UMPATCHER:
umpatcher.py itself is becoming complex enough that small line-based patches can
leave behind:
- duplicate geometry code
- half-replaced layout blocks
- stale footer logic
- tiny modal leftovers
- overflow/clipping remnants
- misleading success states

Section-based patching would make confirm/apply screen fixes much safer because
entire layout regions could be replaced cleanly instead of stitched together by
many fragile anchors.

HUMAN BENEFIT:
Section-based patching is easier to reason about.
The user can think in chunks:
- replace footer section
- update confirm modal section
- swap backup logic section
- refresh startup self-check section

AI BENEFIT:
AI-generated patches often make assumptions about exact line placement.
Section-based targeting reduces the chance of:
- false matches
- accidental nearby matches
- patch drift
- formatting-dependent failure
- apparent success without real behavioral change

RECOMMENDED SUPPORTING RULES:
- require unique section names where possible
- allow preview of extracted target section before apply
- show before/after diff for the section
- log validation results in sidecar
- record section name, bytes changed, and verification outcome
- warn if user is patching umpatcher.py itself
- prefer section operations over raw anchor operations whenever possible

IDEAL SUCCESS STATES:
- VERIFIED_SECTION_SUCCESS
- SECTION_NOT_FOUND
- SECTION_AMBIGUOUS
- SECTION_BOUNDARY_BROKEN
- SECTION_WRITE_ONLY_WARNING
- SECTION_VALIDATION_FAILED

DESIGN PHILOSOPHY:
Patch the file by meaning, not by accident.

This means the patcher should increasingly understand:
- what block it is changing
- where that block begins and ends
- whether the block truly became what was intended
- whether duplicate leftovers remain afterward

-------------------------------------------------------------------------------
THINKTANK / CONCLUSION
-------------------------------------------------------------------------------

A mature patcher should behave less like a blind text replacer and more like a
cautious file surgeon:
- inspect before
- patch carefully
- inspect after
- prove the result
- warn if uncertain

Preferred long-term direction:
- move away from fragile single-line anchor logic
- move toward section-based patching
- validate exact live-file results after apply
- centralize backups and validation reporting
- record patch verification details in sidecar notation/logging

CONCLUSION:
Patch success must not be based on trust.
Patch success must be based on verified live-file evidence.

Section-based patching is the preferred path forward because it is more
trustworthy, easier to validate, easier to reason about, and much safer for
AI-assisted evolving Ultramode files.

ULTRAMODE NOTATION BLOCK
CATEGORY: PATCHING_ARCHITECTURE
TYPE: DUPLICATE_PATCH_PREVENTION_RULESET
STATUS: ACTIVE IDEA / STRONG RECOMMENDATION
PRIORITY: VERY HIGH
===============================================================================

TITLE:
UMPATCHER MUST PREVENT DUPLICATE PATCHING, REPEATED INSERTS, AND FAKE RE-APPLIES

CORE GOAL:
UMPatcher should not merely ask, "Can I write this patch?"
It should ask, "Would this patch create a real new valid change, or would it
just repeat work, duplicate content, or create patch bloat?"

This means duplicate-patch prevention should become a first-class rule of the
patching engine.

CORE PRINCIPLE:
Do not patch based on write possibility alone.
Patch based on change validity.

A patch should be blocked if it would:
- insert content that already exists
- append content that already exists
- replace content that is already in the desired final state
- create duplicate blocks
- create duplicate section content
- produce no meaningful change
- repeat a previously completed patch result

PROBLEM THIS SOLVES:
Without duplicate-patch prevention, UMPatcher risks:
- repeated inserts
- repeated appends
- duplicate geometry blocks
- duplicate notation blocks
- duplicate UI sections
- fake "success" states
- patch bloat
- confusion about what actually changed
- harder cleanup later
- lower trust in patch results

EVOLVED THINKING:
Duplicate prevention should not rely on one simple check only.
It should use layered logic.

RECOMMENDED LAYERS:

-------------------------------------------------------------------------------
LAYER 1: PRE-APPLY LIVE CONTENT CHECK
-------------------------------------------------------------------------------

Before applying a patch, UMPatcher should inspect the live target file and ask:

- Does the exact intended new block already exist?
- Does the exact intended final section already exist?
- Is the old target text already gone?
- Has this file effectively already been patched to the desired state?
- Would this operation create a duplicate of content already present?

If the answer shows the patch result already exists, do not apply again.

GOOD OUTCOMES FOR THIS LAYER:
- ALREADY_PATCHED
- NO_MEANINGFUL_CHANGE
- DUPLICATE_BLOCK_DETECTED

IMPORTANT NOTE:
This check should be content-aware, not just timestamp-aware or size-aware.

-------------------------------------------------------------------------------
LAYER 2: PATCH FINGERPRINT / IDENTITY
-------------------------------------------------------------------------------

Each patch should have a stable identity or fingerprint.

Possible fingerprint ingredients:
- patch type
- target file
- target section
- target anchor or target pattern
- normalized intended new content
- optional version/revision metadata
- optional patch title or patch name

PURPOSE:
If the same logical patch is being applied again against a target that already
reflects that result, UMPatcher should recognize this and refuse duplicate apply.

FUTURE IDEA:
UMPatcher could store fingerprint history:
- in memory during the current run
- optionally in a future sidecar history or patch history system
- optionally in per-file patch receipts later

IMPORTANT CLARIFICATION:
Fingerprinting alone is not enough.
A patch with the same fingerprint may be valid later if the target file changed
again. Therefore fingerprints should be compared along with live target state,
not treated as blind permanent bans.

-------------------------------------------------------------------------------
LAYER 3: TARGET-STATE AWARENESS
-------------------------------------------------------------------------------

A smarter patcher should understand not just the patch, but the target state.

It should reason like this:
- old state expected?
- current state actual?
- final state intended?
- does current already match final?

This is especially important for:
- REPLACE
- FULL
- REPLACE_SECTION
- INSERT_BEFORE
- INSERT_AFTER
- APPEND

EXAMPLES:
1. REPLACE:
   If the replacement text is already present and the old text is absent,
   return ALREADY_PATCHED instead of replacing again.

2. APPEND:
   If the append block already exists in the intended end zone or elsewhere in a
   way that proves it was already added, block duplicate append.

3. INSERT_BEFORE / INSERT_AFTER:
   If the inserted block is already adjacent to the anchor in the intended
   location, do not insert again.

4. REPLACE_SECTION:
   If the section already matches the exact intended final section result,
   return ALREADY_PATCHED or NO_MEANINGFUL_CHANGE.

-------------------------------------------------------------------------------
LAYER 4: POST-APPLY VERIFICATION AGAINST DUPLICATION
-------------------------------------------------------------------------------

After writing, UMPatcher should re-read the file and verify not only that the
new content exists, but that it does not now exist twice by mistake.

Post-apply questions:
- Does the new block exist exactly once where intended?
- Did the old block change as expected?
- Was duplicate content accidentally created?
- Did the section become correct, or bloated?
- Did append/insert create repeated content?

IMPORTANT RULE:
A patch should not be declared VERIFIED_SUCCESS if the result contains duplicate
copies of the inserted/replaced content.

GOOD OUTCOMES FOR THIS LAYER:
- VERIFIED_SUCCESS
- DUPLICATE_CREATED_WARNING
- VALIDATION_FAILED
- WRITE_ONLY_WARNING

-------------------------------------------------------------------------------
LAYER 5: PATCH-TYPE-SPECIFIC DUPLICATE RULES
-------------------------------------------------------------------------------

Different patch types need different duplicate logic.

APPEND RULE:
- if exact append block already exists in the target file, do not append again
- if append is intended only once, detect prior existence and block repeat

INSERT_BEFORE / INSERT_AFTER RULE:
- if intended inserted block is already adjacent to the anchor, do not reinsert
- if anchor exists multiple times, require stricter validation

REPLACE RULE:
- if new text already exists and old text is absent, report ALREADY_PATCHED

FULL RULE:
- if file already matches intended full content, report NO_MEANINGFUL_CHANGE

REPLACE_SECTION RULE:
- if target section already equals intended final section, block re-apply
- if replacement would duplicate nested section content, stop and warn

NOTATION / LOG RULE:
- if the exact notation block already exists, do not append a second identical
  copy unless the user explicitly wants deliberate repetition

-------------------------------------------------------------------------------
SMARTER DECISION MODEL
-------------------------------------------------------------------------------

UMPatcher should think in this order:

1. Can I find the target?
2. What is the file's current state?
3. Does the intended final state already exist?
4. Would applying this patch create a real improvement?
5. Would applying it create duplicate content?
6. If applied, can I verify a clean singular result?

This is better than:
- match line
- write text
- say success

-------------------------------------------------------------------------------
RECOMMENDED STATUS LANGUAGE
-------------------------------------------------------------------------------

Future status results should include:

- ALREADY_PATCHED
  Intended final state already exists.

- NO_MEANINGFUL_CHANGE
  Patch would not materially change the file.

- DUPLICATE_BLOCK_DETECTED
  Patch content already exists and repeating it would create duplication.

- DUPLICATE_BLOCKED
  Duplicate patch attempt intentionally blocked.

- VERIFIED_SUCCESS
  Patch applied and verified cleanly.

- DUPLICATE_CREATED_WARNING
  Write occurred but duplicate content was detected afterward.

- VALIDATION_FAILED
  Patch attempt did not produce the intended clean final state.

-------------------------------------------------------------------------------
AI GUIDANCE
-------------------------------------------------------------------------------

All AI working on this project should treat duplicate prevention as a design
requirement, not a bonus feature.

AI should not assume that:
- matching the target means patching is still needed
- successful write means successful patch
- repeated append/insert is harmless
- duplicate notation is acceptable by default
- duplicate UI blocks are easy to clean up later

AI should instead think:
- is this patch already effectively present?
- would this create duplicate content?
- is the final intended state already achieved?

-------------------------------------------------------------------------------
WHY THIS IS IMPORTANT FOR UMPATCHER
-------------------------------------------------------------------------------

UMPatcher is being used in an evolving AI-assisted workflow.
That makes duplicate-prevention especially important because:
- multiple AI may propose similar patches
- live files may already contain partial or previous versions of the same idea
- repeated testing may accidentally reapply the same patch
- UI/layout blocks are especially vulnerable to duplicate buildup
- logs and notation can bloat if the same block is appended repeatedly

-------------------------------------------------------------------------------
THINKTANK NOTE
-------------------------------------------------------------------------------

A mature patcher should behave less like a blind patch launcher and more like a
state-aware file surgeon.

It should know the difference between:
- "I can write"
and
- "I should write"

That distinction is the heart of duplicate-patch prevention.

-------------------------------------------------------------------------------
CHANGELOG / FIX RULE
-------------------------------------------------------------------------------

If duplicate-prevention logic is later implemented in working code, the change
should be recorded in:
1. changelog / revision notes
2. notation / thinktank history

That record should mention:
- what duplicate scenarios are now detected
- which patch types gained duplicate blocking
- whether fingerprints were added
- whether target-state comparison was added
- whether post-apply duplicate verification was added

CONCLUSION:
UMPatcher should evolve toward state-aware duplicate prevention.

The correct question is not:
"Can this patch be applied again?"

The correct question is:
"Would applying this patch now create a real valid change, or just duplicate
what already exists?"
